# DatePass

DatePass is a romantic, geeky invitation delivered as a signed Apple Wallet boarding pass. The backend is fully serverless: API Gateway, AWS Lambda, DynamoDB, private S3, Secrets Manager, CloudWatch and optional Route 53 custom-domain wiring.

## Architecture

```text
iPhone / Browser
      |
API Gateway custom domain (optional Route 53)
      |
AWS Lambda container (Python 3.12 + FastAPI + Mangum)
      |--- DynamoDB: DatePassInvitations
      |--- S3 private bucket: passes/{id}/datepass.pkpass
      |--- Secrets Manager: datepass/apple-wallet
      |--- EventBridge: DatePassEvents
      `--- CloudWatch Logs
```

The stable shareable URL is `/pass/{id}`. It redirects to a short-lived S3 presigned URL, so the bucket stays private.

## Security design

- Invitation creation requires `X-DatePass-Creator-Key`.
- IDs are random UUIDv4 values.
- The QR opens `/accept/{id}` as a review page. A separate `POST /api/respond/{id}` records the decision, preventing accidental state changes by link-preview bots.
- Wallet certificates and keys are stored only in Secrets Manager.
- The S3 bucket blocks public access, enforces TLS, encrypts objects, retains versions and returns only expiring presigned URLs.
- DynamoDB uses on-demand capacity, AWS-managed encryption and point-in-time recovery.
- API Gateway and Lambda write logs to CloudWatch.
- Accept/decline actions publish an `InvitationResponded` event to EventBridge for future SNS, SES or webhook notifications.

## API

| Method | Path | Purpose |
|---|---|---|
| `POST` | `/invite` | Create invitation, build signed pass and store it in S3 |
| `GET` | `/pass/{id}` | Redirect to a short-lived `.pkpass` download URL |
| `GET` | `/accept/{id}` | Render a confirmation screen |
| `GET` | `/decline/{id}` | Render a rejection screen |
| `POST` | `/api/respond/{id}` | Record accept/decline, regenerate pass and overwrite S3 object |
| `GET` | `/status/{id}` | Render status page in a browser or return JSON to API clients |
| `GET` | `/api/status/{id}` | Always return invitation status as JSON |
| `GET` | `/health` | Health check |

## Apple Developer setup

1. Join the Apple Developer Program using the Apple account that will own DatePass.
2. In Certificates, Identifiers & Profiles, create a Pass Type ID such as `pass.com.example.datepass`.
3. Create a Pass Type ID certificate for that identifier. Export the certificate and private key from Keychain Access as a `.p12` file.
4. Download Apple’s current WWDR intermediate certificate from Apple PKI.
5. Convert the files locally. Do not commit any certificate or private key.

```bash
openssl pkcs12 -in datepass.p12 -clcerts -nokeys -out pass-cert.pem
openssl pkcs12 -in datepass.p12 -nocerts -out pass-key-encrypted.pem
openssl x509 -inform DER -in AppleWWDRCAG4.cer -out wwdr.pem
```

6. Generate the Secrets Manager JSON payload:

```bash
python3 scripts/create_secret_payload.py \
  --pass-type-id pass.com.example.datepass \
  --team-id YOUR_TEAM_ID \
  --organization DatePass \
  --certificate pass-cert.pem \
  --private-key pass-key-encrypted.pem \
  --wwdr-certificate wwdr.pem \
  --private-key-password 'your-local-passphrase' \
  --creator-api-key "$(openssl rand -hex 32)" > secret.json
```

## Deployment

Prerequisites: AWS CLI v2, Docker, Node.js, AWS CDK CLI and Python 3.12.

```bash
npm install -g aws-cdk
python3 -m venv .venv
source .venv/bin/activate
pip install -r infra/requirements.txt
aws configure
cdk bootstrap
cdk deploy
```

Without a custom domain, CDK injects the generated API Gateway stage URL into the Lambda automatically. For a custom domain, export `DATEPASS_API_BASE_URL=https://date.example.com` before deployment so links embedded inside Wallet passes use that domain.

Store the secret payload after the first deployment:

```bash
aws secretsmanager put-secret-value \
  --secret-id datepass/apple-wallet \
  --secret-string file://secret.json
rm secret.json
```

Run a smoke test:

```bash
export DATEPASS_URL='https://abc123.execute-api.us-east-1.amazonaws.com/prod'
export DATEPASS_CREATOR_KEY='copy-the-generated-creator-key'
./scripts/smoke_test.sh
```

For a custom domain, create or import an ACM certificate in `us-east-1` for an edge-optimized REST API and export the optional variables shown in `.env.example` before deployment. The stack creates the API Gateway domain mapping and Route 53 alias.

## Pass package

Each `.pkpass` generated by `app/wallet_pass.py` contains:

```text
pass.json
icon.png
icon@2x.png
logo.png
logo@2x.png
manifest.json
signature
```

`manifest.json` hashes each pass source file. `signature` is a detached PKCS #7 signature containing the Apple WWDR intermediate certificate. The `.pkpass` file is a ZIP bundle with no parent directory.

## Updating the pass

Version 1 regenerates and overwrites `passes/{id}/datepass.pkpass` after acceptance or rejection. The recipient can download the refreshed pass from the stable `/pass/{id}` URL.

A production phase 2 can implement Apple Wallet push updates by adding `webServiceURL` and `authenticationToken` to `pass.json`, storing device registrations and push tokens, implementing Apple’s pass web-service endpoints and sending APNs notifications when the pass changes.

## Notes for production hardening

- Replace creator-key authentication with Cognito or IAM authorizers if multiple creators will use the service.
- Add AWS WAF rate limiting at the API edge for a public launch.
- Attach an EventBridge rule and SNS, SES or webhook target to `DatePassEvents` for creator notifications.
- Add an expiration policy for old S3 object versions and archived invitations.
- Add a CSRF token if you expose additional browser forms or authenticate recipients with cookies.
- Rotate the creator key and Apple certificates with an operational runbook.
